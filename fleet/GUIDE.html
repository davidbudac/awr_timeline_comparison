AWR Fleet Warehouse

Deployment & usage guide — collect AWR history from a fleet of Oracle 19c databases into one central warehouse, analyze the whole fleet on a schedule, and get notified only about the databases where something actually changed.

Pure-SQL / PL​SQL toolkit · read-only on every source · single self-contained HTML report for drill-down

This guide walks from zero to a running 50-database deployment, then covers day-2 operations, tuning, and how to make a large rollout painless.

1 · What this is & the big picture

The toolkit has two halves that work together:

  • The single-database report (repo root: run_awr_trend.sh, awr_trend.sql) — compares the same hour across periods for one database and renders a self-contained HTML report. This is your drill-down.
  • The fleet warehouse (fleet/) — pulls AWR from many databases into one central 19c warehouse, scores the whole fleet, and emits a digest that surfaces only the interesting databases. This is what you deploy across 50 DBs.

The defining design choice: sources are only ever read. The collector is INSERT … SELECT@dblink, so each monitored database needs nothing but a SELECT-only account. The warehouse is the only thing that writes, and it recomputes everything from raw DBA_HIST_* on each run — so it works against a physical standby and never installs objects on your production databases.

2 · Core concepts (10-minute model)

TermWhat it means
TargetOne monitored database — the unit of identity. Stable; never derived from DBID or name. Reached through exactly one read-only DB link to one container (a PDB or non-CDB, never a CDB root).
DBID setThe DBIDs a Target owns. Auto-discovered through the link on first collection — including an old DBID kept after a non-CDB→PDB migration. A DBID already owned by another Target is logged as a collision, never silently merged.
Metric ProfileThe named scoring config: z-score thresholds, baseline depth (weeks_back), window width, cadence, hysteresis, and the per-metric noise gates. One DEFAULT profile ships; you can add more and assign per Target.
FindingA notable observation about one subject (a metric / SQL / wait event / segment) on one Target at one Period. Append-only history.
Alert StatePer-subject lifecycle that collapses a run of Findings into one fire and one clear, with hysteresis (must be anomalous N periods to fire, normal M to clear) so nothing flaps.
DigestThe periodic HTML roll-up: the interesting (firing) databases, headline movers, recoveries, and coverage gaps.

The three detectors

Each analysis Period runs all three over the same comparison windows, in one transaction:

  • Seasonal — z-scores a curated metric set (load / system-metric / wait classes) against the same hour in prior weeks; gated by magnitude, percent, and direction → fires alerts.
  • Regression — the top SQL / wait events / segments whose current impact regressed versus their own baseline, ranked by impact, with SQL plan flips flagged → fires alerts.
  • Headline — the marquee “hero-six” as an executive snapshot (ungated, informational) → shown as headline movers in the digest.
Why seasonal, not thresholds? A database that always runs at 80% CPU at 9am Monday isn't an incident — it's Monday. Comparing the current hour to the same hour in prior weeks catches genuine change and ignores normal daily/weekly rhythm, with no per-database thresholds to maintain.

3 · How one cycle works

One job runs the whole fleet cycle. Each phase is isolated — a single unreachable database or bad Target never aborts the run — and every phase is bookended in awrw_run_log.

Findings are stored, Alert State advances (fire/clear with hysteresis), and the digest is both archived to awrw_digest and written to disk. The very first time a database is analyzed there isn't enough same-hour history yet, so you'll see INSUFFICIENT_HISTORY until weeks_back + 1 weeks of that hour have accrued — that's expected, not a fault.

4 · Prerequisites

Warehouse (one central database)

  • A dedicated Oracle 19c database or PDB. Sizing is modest — the facts are a thin extract, not a full AWR copy.
  • An owner schema (e.g. AWRWH) with: CREATE SESSION, CREATE TABLE, CREATE TYPE, CREATE VIEW, CREATE PROCEDURE, CREATE DATABASE LINK, CREATE JOB (for the scheduler), and a tablespace quota.
  • For filesystem delivery only: a one-time CREATE ANY DIRECTORY by a privileged user (see step 9).

Each source (×50)

  • Oracle 19c with the Diagnostic & Tuning Pack licensed (AWR requires it).
  • AWR snapshots actually being taken (the stock hourly job is fine).
  • Network reachability from the warehouse to the source listener (typically 1521).
  • A read-only reader account (created in step 2).

5 · Deploying across 50 databases — walkthrough

1Install the warehouse

As the warehouse owner, run the master installer once. It creates the schema, the AWRV_* seam, all packages, and seeds the DEFAULT profile.

-- on the central warehouse, as e.g. AWRWH
cd fleet
sqlplus awrwh/pw@warehouse @install_warehouse.sql

It finishes by listing every object's status — confirm they're all VALID.

2Create a read-only reader on each source

Run in the monitored container (the PDB or non-CDB — never the CDB root). Least privilege:

CREATE USER awr_reader IDENTIFIED BY "<password>";
GRANT CREATE SESSION TO awr_reader;
GRANT SELECT_CATALOG_ROLE TO awr_reader;   -- covers the DBA_HIST_* views
Standardize it. Use the same reader name and password on all 50 databases. That collapses 50 bespoke scripts into one script your DBAs (or Ansible) run everywhere — and makes the bootstrap loop in §6 trivial.

3Create the DB link (warehouse side)

One private link per Target, pointing at the source container's service. EZConnect avoids maintaining tnsnames.ora:

CREATE DATABASE LINK prod01
   CONNECT TO awr_reader IDENTIFIED BY "<password>"
   USING '//host01:1521/PROD01_SVC';

-- sanity check the link works and the reader can see AWR:
SELECT count(*) FROM dba_hist_snapshot@prod01;

4Register the Target

BEGIN
  awrw_admin.add_target(p_name => 'PROD01', p_db_link => 'PROD01',
                        p_snap_interval_min => 60);
END;
/

You don't enter DBIDs — the DBID set is auto-discovered on first collection. Re-running add_target for the same name simply updates the registry (it's an upsert).

5First collection — backfill

EXEC awrw_collect.collect_all;        -- every enabled Target
EXEC awrw_collect.refresh_health;     -- compute CURRENT/LAGGING/STALE

The first run backfills whatever AWR each source still retains (high-water mark starts at 0); later runs pull only new snapshots. Collection is snapshot-atomic per DBID — all facts for a range commit together and the high-water mark only advances on success, so a partial pull just re-runs next cycle.

6Verify coverage, then analyze

-- who's current vs lagging?
SELECT t.target_name, h.collect_status, h.last_snap_end_ts, h.consecutive_fail
  FROM awrw_target t JOIN awrw_target_health h ON h.target_id = t.target_id
 ORDER BY h.collect_status, t.target_name;

-- anything erroring? (e.g. a link down, missing Diagnostic Pack)
SELECT * FROM awrw_error_log ORDER BY err_ts DESC FETCH FIRST 50 ROWS ONLY;

EXEC awrw_analyze.analyze_all;        -- score every Target
SELECT awrw_notify.build_digest FROM dual;   -- preview the HTML (read-only)

7Schedule the whole cycle

EXEC awrw_schedule.create_jobs;                         -- hourly at :05
-- or any cadence:
EXEC awrw_schedule.create_jobs('FREQ=HOURLY;BYMINUTE=30');

That single AWRW_PIPELINE job runs collect → health → analyze → digest for the entire fleet. No per-database jobs.

8Turn on a delivery method

Filesystem (built in). Enable it once by creating the target directory as a privileged user (edit the path and owner in the template), then it just works every cycle:

-- as a privileged user, after editing fleet/schedule/digest_dir.sql
@@schedule/digest_dir.sql
-- each cycle now drops awr_fleet_digest_<id>_<ts>.html + awr_fleet_digest_latest.html

Email / Slack (seam). Site-specific, so left to you — drain the archive and send:

SELECT digest_id, html FROM awrw_digest WHERE delivered_ts IS NULL ORDER BY digest_id;
-- ...send html as an HTML email (UTL_SMTP needs a network ACL) or hand to a mailer...
UPDATE awrw_digest SET delivered_ts = systimestamp WHERE digest_id = :id;

6 · Simplifying a 50-database rollout

Steps 2–4 are the only ones that repeat per database. The trick is to make them data-driven instead of 50 manual sequences. Here's the answer to “what could simplify this?”, strongest first.

a) Bootstrap from a driver table biggest win

List the fleet in one table, then loop once: create each link and register each Target in a single run. The DBID discovery, collection, analysis, scheduling, and delivery are already fleet-wide single calls — so this is the only thing left to batch.

-- 1. describe the fleet (one row per database)
CREATE TABLE awrw_bootstrap (
  name    VARCHAR2(128),  host VARCHAR2(255), port NUMBER,
  service VARCHAR2(128),  reader VARCHAR2(128), pw VARCHAR2(128));

INSERT INTO awrw_bootstrap VALUES ('PROD01','host01',1521,'PROD01_SVC','awr_reader','<pw>');
-- ... 49 more rows (or load from a CSV) ...
COMMIT;

-- 2. one loop builds every link + registers every Target
BEGIN
  FOR r IN (SELECT * FROM awrw_bootstrap) LOOP
    BEGIN EXECUTE IMMEDIATE 'DROP DATABASE LINK '||r.name; EXCEPTION WHEN OTHERS THEN NULL; END;
    EXECUTE IMMEDIATE
      'CREATE DATABASE LINK '||r.name||
      ' CONNECT TO '||r.reader||' IDENTIFIED BY "'||r.pw||'"'||
      ' USING ''//'||r.host||':'||r.port||'/'||r.service||'''';
    awrw_admin.add_target(p_name => r.name, p_db_link => r.name);
  END LOOP;
END;
/

DROP TABLE awrw_bootstrap PURGE;   -- it held passwords — drop it after
Credential hygiene. That table holds plaintext passwords — drop it immediately after, and prefer a standardized reader account so it's one secret, not fifty. Better still, use an Oracle Wallet / external password store so the links carry no plaintext and rotation is central (then the loop omits IDENTIFIED BY).

b) Standardize the source side

One reader name + password across the fleet means step 2 is a single script your existing config management (Ansible / Salt / a deployment pipeline) pushes to all 50 — no per-database editing. Same idea for EZConnect: //host:port/service in the link removes all tnsnames.ora upkeep.

c) Preflight before you enable

Catch the usual rollout problems (link down, firewall, missing Diagnostic Pack, wrong service) in one sweep before scheduling:

DECLARE n NUMBER; BEGIN
  FOR t IN (SELECT target_name, db_link FROM awrw_target WHERE enabled='Y') LOOP
    BEGIN
      EXECUTE IMMEDIATE 'SELECT count(*) FROM dba_hist_snapshot@'||t.db_link INTO n;
      dbms_output.put_line(rpad(t.target_name,20)||' OK   snaps='||n);
    EXCEPTION WHEN OTHERS THEN
      dbms_output.put_line(rpad(t.target_name,20)||' FAIL '||sqlerrm);
    END;
  END LOOP;
END;
/

d) What's already fleet-scale (no work needed)

  • Discovery — DBIDs (including post-migration old ones) are found automatically.
  • Collection / analysis / digest — single *_all calls cover every Target.
  • Scheduling — one job for the whole fleet, not 50.
  • Coverage — one query over awrw_target_health shows all 50 at a glance, so silence can never be mistaken for health.
Possible future helper. An awrw_admin.add_target overload that also creates the link in one call would fold steps 3–4 into one — say so if you'd like it added (it's a small, testable addition).

7 · Operating the fleet (day 2)

What's happening right now

-- freshness per Target
SELECT t.target_name, h.collect_status, h.last_snap_end_ts, h.consecutive_fail
  FROM awrw_target t JOIN awrw_target_health h ON h.target_id=t.target_id
 ORDER BY h.collect_status, t.target_name;

-- recent cycles and errors
SELECT * FROM awrw_run_log   ORDER BY started_ts DESC FETCH FIRST 50 ROWS ONLY;
SELECT * FROM awrw_error_log ORDER BY err_ts     DESC FETCH FIRST 50 ROWS ONLY;

-- what's firing across the fleet
SELECT t.target_name, a.subject_type, a.subject_id, a.severity, a.fired_period_ts
  FROM awrw_alert_state a JOIN awrw_target t ON t.target_id=a.target_id
 WHERE a.state='FIRING' ORDER BY a.severity, t.target_name;

Coverage status meanings

StatusMeaning
CURRENTCollecting on cadence — healthy.
LAGGINGBehind expected, but recent — usually transient.
STALENo fresh snapshots for a while — investigate.
UNREACHABLEThe link/collection is failing — check awrw_error_log.

Retention

Facts carry snap_day; in production, RANGE/INTERVAL-partition the fact tables on it and purge by dropping partitions (never below weeks_back + 1 weeks). awrw_findings and awrw_alert_state are append-style — schedule a purge of old findings and of NORMAL alert rows that haven't moved in weeks_back weeks, alongside the analyze job.

8 · Tuning what fires

All tuning lives in the Metric Profile — no code changes. The header (awrw_profile) sets the shape and thresholds; the per-metric rows (awrw_profile_metric) set the noise gates.

KnobDefaultEffect
z_crit / z_warn3 / 2|z| over these → CRITICAL / WARN.
weeks_back4How many prior same-hour windows form the baseline.
step_days7Cadence between windows (7 = same hour-of-week).
pct_floor25Minimum % change to alert (kills tiny-but-significant noise).
fire_consec / clear_consec2 / 3Hysteresis: periods anomalous to fire / normal to clear.
reg_top_n10Max regression findings per subject type per period.
reg_sql_min_sec / reg_wait_min_sec / reg_seg_min_blk5 / 30 / 100kMagnitude floors so trivial SQL / waits / segments don't fire.
-- example: a noisier database gets a stricter profile
UPDATE awrw_profile SET z_warn=2.5, pct_floor=40 WHERE profile_name='DEFAULT';
-- ...or create a second profile and assign it to specific Targets via add_target(p_profile=>...)

9 · Drilling down on one database

When the digest flags a database, render the full single-database report for it — the same seasonal comparison, but with every chart and table, run live against the source (or against the warehouse):

./run_awr_trend.sh user/pw@prod01 '2026-06-08 10:00' 1 4 25
--                  service          target hour      win weeks topN

It produces one self-contained HTML file (charts degrade gracefully offline). The fleet digest tells you where to look; this report tells you what happened.
