The database did not behave the way it did on the last four Mondays. Total DB time for the hour was 4.6 hours against a baseline of 2.9, and the growth is concentrated in commit waits and one SQL statement introduced with the release.

What changed

1

Commit latency tripled. Average log file sync wait rose from 4.1 ms to 14.2 ms (+3.4σ) and now accounts for 64% of DB time. Consistent with a new synchronous commit pattern, not with storage degradation — log file parallel write is flat at 0.7 ms.

log file sync 14.2 ms +3.4σ ████████████▌ 64% log file par write 0.7 ms +0.1σ unchanged
2

One statement is doing three times the work. 7k2m9x4fqz1nc (module ORDERS_API, schema ORDERS) went from ~380 s to 1,204 s of elapsed time (+212%). First seen in this shape on Sunday's release.

7k2m9x4fqz1nc ela 1,204 s execs 214k +212% vs base
3

Throughput is up, not just latency. Executes/s rose 28% (+2.1σ). Some of the extra DB time is legitimate new load; findings 1–2 are not explained by it.

What did not change

I/O latencies, background waits, memory and parameter set are all within ±1σ of baseline. No instance restarts in any window. 47 of 52 tracked metrics: quiet.
